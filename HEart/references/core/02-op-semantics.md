# CKKS Operation Semantics

Use this as an engineering instruction set. The exact API names differ by library, but these state transitions are the CKKS-level behavior to track: `level`, `scale`, `noise/error`, and required keys.

Global cost order: `Add ~= AddPlain << MulConst < Rescale/DropLevel < Relinearize <= Mul ~= Rotate ~= Conjugate << Bootstrap`. Treat this as a planning heuristic; concrete costs depend on `N`, number of RNS limbs, key-switch decomposition, rotation count, and implementation. [2016-421][2018-931][2020-1203][2021-204]

| Primitive | Preconditions: level / scale alignment | Scale effect | Level / depth effect | Noise effect | Required key | Relative cost |
|---|---|---|---|---|---|---|
| Add [2016-421][2018-931][2020-1203] | Operands must be at a common usable modulus; scales must match or be aligned to the larger scale. | Output scale is the aligned/max scale. | No multiplicative depth; output uses the shared smaller modulus if levels differ. | Adds errors; usually small compared with multiplication if scales are sane. | None. | Low. |
| AddPlain [2016-421][2018-931][2020-1203] | Plaintext must be encoded at a compatible scale and level with the ciphertext. | Output scale is the aligned/max scale. | No multiplicative depth; may drop to shared modulus. | Adds plaintext encoding error plus ciphertext error. | None. | Low. |
| MulConst / MulPlain [2016-421][2018-931][2020-1203] | Constant/plaintext must be encoded for the intended scale and level. | Multiplies scale by the constant/plaintext scale; in ordinary CKKS plaintext-ciphertext multiplication, follow with rescale to restore the working scale. | Raw multiply consumes no level; the required follow-up rescale consumes one level. | Multiplies existing error by constant/plaintext magnitude and adds encoding/rounding error. | None. | Low to medium. |
| Mul [2016-421][2018-931][2020-1203] | Ciphertexts must share a usable modulus; if levels differ, first `DropLevel`/mod-switch the higher-level operand to the lower shared level; scales should be planned because output scale is the product. | Output scale is approximately `Delta1 * Delta2`; follow with rescale to restore the working scale. | Raw multiplication increases ciphertext degree; normal pipeline then relinearizes and rescales, spending one level. | Largest basic noise growth: product error plus key-switch error if relinearized. | Relinearization key for normal 2-component output. | High. |
| Relinearize [2020-1203][2021-204] | Input is an extended ciphertext, typically degree 2 after `Mul`. | Scale unchanged. | Level unchanged at the CKKS interface, though key switching uses decomposition/auxiliary moduli internally. | Adds key-switching error. | Relinearization / switching key. | Medium to high. |
| Rescale / ModReduce [2016-421][2018-931][2020-1203] | Level must be above zero; usually applied when scale is near `Delta * q_l` or otherwise scheduled to divide by the removed prime. | Divides scale by the dropped prime `q_l`; target is usually back near `Delta`. | Drops one chain prime and spends one multiplicative-depth level. | Adds rounding/rescale error; also reduces the magnitude of message and prior error. | None. | Medium. |
| Rotate [2016-421][2018-931][2020-1203] | Rotation step must be supported by a generated Galois/rotation key; level and scale otherwise unchanged. | Scale unchanged. | Level unchanged. | Adds key-switching error. | Rotation key for that step or a generated basis that composes it. | High. |
| Conjugate [2018-153][2020-1203] | Conjugation key must be generated; required for complex conjugation and real/imag extraction patterns. | Scale unchanged. | Level unchanged. | Adds key-switching error. | Conjugation Galois key. | High. |
| ModSwitch / DropLevel [2012-099][2020-1203][2021-204] | Target must be a lower modulus/level; use only when scale semantics should remain unchanged, unlike rescale. | Scale unchanged for CKKS `DropLevel`. | Drops one or more primes without claiming multiplicative-depth normalization. | May add/modulate rounding noise depending on scheme and implementation; mainly used for level alignment. | None. | Low to medium. |
| Bootstrap [2018-153][2019-688][2020-1203][2020-552] | Input must satisfy the bootstrap parameter range, residual precision, scale, and slot-layout assumptions; bootstrap keys must cover required linear transforms. | Produces a controlled post-bootstrap scale, not an exact reset; scale choices affect precision. | Consumes many internal levels but returns a ciphertext with refreshed usable modulus/depth. | Replaces accumulated error with bootstrap approximation, linear-transform, key-switch, and rescale errors. | Bootstrapping key set: many CoeffToSlot/SlotToCoeff rotation keys, conjugation, relinearization/switching material, and scheme-specific setup. | Very high. |

## Engineering Notes

- Add: align levels before adding a high-level ciphertext to a low-level one; otherwise the high-level operand's extra primes are discarded at the shared modulus. Use this deliberately for fan-in sums. [2016-421][2020-1203]
- AddPlain: encode plaintext constants once per intended scale/level when possible; a constant encoded at the wrong scale is a numerical bug, not a type bug. [2016-421][2020-1203]
- MulConst / MulPlain: small constants are cheap, but large-magnitude constants amplify error and still require rescale when the plaintext is encoded with a nontrivial CKKS scale. Treat constant/plaintext magnitude as part of the noise budget. [2016-421][2020-1203]
- Mul: budget multiplication as a bundle: ciphertext product, relinearization, and rescale. Counting only the algebraic multiply underestimates time, key requirements, and level use. Before another multiplication, relinearize unless the library explicitly supports lazy relinearization; a common lazy pattern is to add several degree-2 products first and relinearize once. Most libraries also expose a specialized square operation (`EvalSquare`, `square`, or similar). [2016-421][2020-1203][openfhe-development-1.5.1/openfhe-development-1.5.1/src/pke/include/schemebase/base-leveledshe.h]
- Relinearize: this is key switching from the multiplication key component back to the normal secret-key form. Hybrid/RNS decomposition parameters trade key size, NTT count, auxiliary modulus size, and noise. [2012-099][2021-204]
- Rescale / ModReduce: in full-RNS CKKS, rescale divides by the next chain prime, so exact power-of-two scale schedules become approximate. Expect small scale deviations and handle them explicitly. Naming hazard: OpenFHE's `ModReduce`/rescale path divides the scale, while `LevelReduce`/`LevelReduceInternal` drops towers without changing the underlying plaintext; SEAL's `rescale_to_next` divides the message/scale, while `mod_switch_to_next` only switches to a smaller modulus. Therefore OpenFHE `ModReduce` is not SEAL `mod_switch_to_next`; check `lib-*.md` before mapping APIs. [2018-931][2019-688][openfhe-development-1.5.1/openfhe-development-1.5.1/src/pke/include/schemebase/base-leveledshe.h][SEAL/native/src/seal/evaluator.h]
- Rotate: every rotation in a diagonal matrix multiply, CoeffToSlot, SlotToCoeff, convolution, or slot-sum must be enumerated before key generation. Missing rotation keys are a design-note failure. [2018-153][2020-1203]
- Conjugate: use it for complex conjugation and real/imag extraction patterns, including bootstrap sine/imaginary-part steps. It is as expensive as a key switch and should be counted. [2018-153][2020-1203]
- ModSwitch / DropLevel: use it to align levels when scale should not be divided, including before multiplication between operands at different levels. Do not substitute it for rescale after ciphertext-ciphertext or plaintext-ciphertext multiplication; those multiplications increase scale and need rescale in the normal CKKS scale schedule. [2016-421][2020-1203][openfhe-development-1.5.1/openfhe-development-1.5.1/src/pke/include/schemebase/base-leveledshe.h][SEAL/native/src/seal/evaluator.h]
- Bootstrap: model it as an approximate circuit: ModRaise, CoeffToSlot, EvalMod/EvalExp or sine/cosine approximation, imaginary/real extraction where applicable, and SlotToCoeff. CoeffToSlot/SlotToCoeff rotation keys are often dominant in key volume and runtime; baby-step/giant-step linear transforms can reduce the rotation count. Its output must be validated slot-wise against a plaintext reference. [2018-153][2019-688][2020-1203][2020-552]
