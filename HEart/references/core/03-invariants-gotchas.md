# CKKS Invariants and Gotchas

Failure-mode format: `[Name] Symptoms -> Cause -> Fix`. Treat each entry as a debugging checklist item, not a paper summary.

| Failure mode | Symptoms -> Cause -> Fix | Sources |
|---|---|---|
| [Scale mismatch] | Symptoms: addition produces visibly worse precision, library rejects operands, or decrypted values drift after an apparently harmless add. Cause: full-RNS rescale divides by chain primes, not exact powers of two, so same-level ciphertexts may carry slightly different scales; adding different scales injects approximation error. Fix: maintain an explicit scale schedule; before addition, align scales by planned rescale/drop-level rules or use a scale-preserving polynomial evaluator. | [2016-421][2018-931][2020-1203] |
| [Level mismatch] | Symptoms: add/mul fails, silently drops modulus limbs, or later multiplication runs out of level unexpectedly. Cause: ciphertexts live under different active moduli `Q_l`; operations use the common smaller modulus, while rescale changes both level and scale. Fix: bring the higher-level operand down intentionally; use DropLevel/mod-switch when scale must be preserved, and Rescale only when the scale should be divided. | [2016-421][2020-1203] |
| [Depth exhausted] | Symptoms: final ciphertext cannot be multiplied/rescaled, bootstrap is required earlier than planned, or decryption has too few reliable bits. Cause: every ciphertext-ciphertext multiplication normally consumes one rescale level, and polynomial/comparison circuits consume levels according to their multiplicative depth. Fix: count `Mul -> Relin -> Rescale` as one depth unit, use Paterson-Stockmeyer/BSGS for high-degree polynomials, reduce degree, increase `N` and chain length if security allows, or insert bootstrap before level 0. | [2016-421][2020-1203][2020-552] |
| [Missing rotation key] | Symptoms: slot sum, convolution, CoeffsToSlots, SlotsToCoeffs, or matrix-vector multiply fails at runtime or requires regeneration of keys. Cause: rotations are Galois automorphisms followed by key switching; each required step needs a rotation key or a library-supported generated basis. Fix: derive all rotation steps from the packing layout before key generation; include BSGS baby and giant steps, conjugation if real/imag extraction is used, and bootstrap linear-transform rotations. | [2016-421][2018-153][2020-1203] |
| [NTT-domain mismatch] | Symptoms: plaintext-ciphertext multiplication gives nonsense, library rejects plaintext form, or performance collapses due to repeated transforms. Cause: full-RNS implementations use double-CRT/RNS plus NTT representation for arithmetic, while some modulus/basis conversions or encoded plaintexts require a specific domain. Fix: track for each plaintext/ciphertext whether it is coefficient-domain or NTT-domain according to the library API; pre-transform static plaintext diagonals/constants once; do not mix domains implicitly. | [2018-931][2022-915] |
| [Imaginary-part leakage / semantic contamination] | Symptoms: a real-only pipeline decrypts with non-negligible imaginary components, comparisons near zero flip, or downstream real outputs drift. Cause: CKKS packs complex slots; conjugate symmetry and real extraction are conventions, not automatic semantic enforcement. This is a numerical/semantic issue, distinct from Li-Micciancio approximate-decryption leakage. Fix: use real-aware packing, check both real and imaginary errors, use conjugation-based real/imag extraction where needed, and keep approximation domains away from discontinuities. | [2016-421][2018-153][2021-1337] |
| [Precision loss after rescale/key-switch] | Symptoms: output error grows faster than predicted, low bits vanish after several multiplications, or bootstrap output has fewer bits than the table promised. Cause: CKKS error includes encoding, encryption noise, key-switch error, rescale rounding, approximation error, and sometimes scale deviation. Fix: validate with plaintext reference after each major block; report max absolute/relative error; for polynomial evaluation, use scale propagation and exact-scale additions; for bootstrap, tune sine/cosine degree, arcsine/composite correction, and linear-transform moduli. | [2016-421][2020-1203][2020-552] |
| [Approximation-domain miss] | Symptoms: sigmoid/ReLU/comparison works on tests but fails on rare production inputs; polynomial output explodes outside the fitted interval. Cause: encrypted intermediate values cannot be inspected, and polynomial approximations are only controlled on their design interval. Fix: bound every nonlinear input range in the design note; for rare outliers, use a larger approximation interval, domain-extension polynomials, or clipping/capping logic that is itself approximated. | [2022-280][2019-1234] |
| [Comparison near zero] | Symptoms: `sign(x)` or `a > b` is unstable when `|a-b|` is tiny. Cause: sign is discontinuous at zero; polynomial comparison papers exclude a small gap `[-epsilon, epsilon]` and guarantee only outside that margin. Fix: specify a comparison margin `epsilon`; treat results inside the gap as indeterminate or use application-level tie handling; increase precision only after checking depth and security. | [2019-1234][2020-834][2021-1337] |
| [Parameter/security violation] | Symptoms: estimator reports below-target security, library rejects parameters, or adding one level forces a much larger ring dimension. Cause: for fixed `N`, the total modulus has a security upper bound; hybrid key switching adds special modulus `P`, so the relevant size can be `log(Q*P)`, not only `log(Q)`. Fix: compute the full modulus budget before coding; if depth does not fit, increase `N`, reduce depth, lower scale, switch evaluation strategy, or bootstrap. | [2016-421][2018-931][2020-1203][2022-915] |
| [Special-prime underbudget] | Symptoms: relinearization/rotation noise is high or key-switch fails under aggressive parameters. Cause: RNS key switching uses special primes/auxiliary basis; too small an auxiliary modulus increases key-switch error, while too large a `P` hurts security. Fix: choose `P` with the key-switch decomposition in mind and include it in the security estimate; prefer library-generated defaults until the key-switch model is documented. | [2018-931][2021-204][2020-1203] |
| [Bootstrap placed too late] | Symptoms: ciphertext reaches level 0 with too much error for EvalMod or cannot satisfy bootstrap input assumptions. Cause: bootstrapping is an approximate circuit with its own input range, scale, level, and precision assumptions; it is not a magic repair for arbitrary invalid ciphertexts. Fix: place bootstrap while enough precision remains; tune `Q0/Delta`, EvalMod interval, slots, and output precision; test the exact pre-bootstrap distribution. | [2018-153][2020-1203][2020-552] |
| [Bootstrap rotation-key explosion] | Symptoms: bootstrap key generation/storage dominates runtime or memory. Cause: CoeffsToSlots and SlotsToCoeffs are large linear transforms requiring many rotations and key switches. Fix: use BSGS/hoisting/double-hoisting where supported; reduce slot count or sparse packing if the application allows it; enumerate the exact rotation set before key generation. | [2018-1073][2020-1203] |
| [CKKS approximate-decryption leakage] | Symptoms: an application returns raw decrypted CKKS outputs to a client/adversary, or exposes many decryptions of evaluated ciphertexts. Cause: approximate decryption reveals `message + error`; Li-Micciancio show passive key-recovery from approximate decryption results, so IND-CPA is not enough when decryptions are exposed. Fix: never expose raw decryption to an untrusted party; use noise flooding / differentially private decryption noise; model the number of adversarial decryption queries. | [2020-1533][2022-816] |
| [Noise flooding overtuned] | Symptoms: security mitigation works but application accuracy collapses. Cause: flooding noise must hide the decryption error distribution, but it directly reduces output precision. Fix: choose flooding from a documented static/dynamic noise estimate; test final accuracy after flooding, not before; record the computational and statistical security targets separately. | [2022-816] |
