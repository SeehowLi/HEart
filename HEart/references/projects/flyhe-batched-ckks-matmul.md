## User-Facing Recommendation

For large batched CKKS matrix multiplication on GPU, start from FlyHE's existing leveled MatMul path before adding bootstrapping or scheme conversion. The local tree already contains a matrix-multiplication experiment:

- Build target: `RLWE/boot/examples/nn.cu` via executable `nn`.
- Kernel implementation: `RLWE/boot/src/matrix_mul.cu` and `RLWE/boot/include/matrix_mul.cuh`.
- Built-in dataset: `RLWE/boot/data/input/matrixmul_input_m_128_n_768_k_64_batch_128.txt`, `RLWE/boot/data/input/matrix_input_n_768_k_64.txt`, and `RLWE/boot/data/calibration/matrix_output_m_128_k_64_batch_128.txt`.
- Baseline shape: `(4096 x 768) * (768 x 64) -> (4096 x 64)`, generated by `RLWE/boot/data/data_generation.py`.

Decision rule: use the existing `MMEvaluator` path when one operand is plaintext/model data and the other is encrypted/compressed, because it uses ciphertext/plaintext products and avoids ciphertext-ciphertext multiplication depth. If both matrices must be encrypted, this implementation is not the right starting point without a new design note for ciphertext-ciphertext products, relin, rescale, and much higher memory pressure.

## CKKS Design Note

### 1. Function, shapes, and range

Function:

```text
f(A, B) = A * B
A: 4096 x 768 plaintext matrix in the current FlyHE example
B: 768 x 64 encrypted/compressed matrix in the current FlyHE example
Output: 4096 x 64 encrypted CKKS result, one ciphertext per output column
```

The bundled generator samples both matrices uniformly in `[-1, 1]`; an output dot product sums 768 products, so a conservative output magnitude bound is about `768` before statistical cancellation. The example compares `mm_res[i] / 2.0` against the plaintext calibration, which is an implementation-specific correction that needs explanation before publication [need manual confirmation]. Sources: `FlyHE-main/FlyHE-main/RLWE/boot/data/data_generation.py`, `FlyHE-main/FlyHE-main/RLWE/boot/examples/nn.cu`.

### 2. Arithmetic circuit

For each output column `c` and row slot `r`:

```text
out_c[r] = sum_{j=0}^{767} A[r,j] * B[j,c]
```

Current FlyHE path:

- `B` is compressed into CKKS ciphertexts.
- Decompression expands compressed ciphertexts into per-coordinate ciphertexts by repeated Galois automorphisms, additions, and coefficient shifts.
- `A[:,j]` is encoded as CKKS plaintext vector.
- Each output column performs 768 `multiply_plain` operations and one `add_many`.
- No ciphertext-ciphertext multiplication is used in the matrix-product accumulation, so no relin is needed for the product itself.

Depth:

- Decompression uses rotations/Galois and additions: depth 0.
- Each plaintext-ciphertext product with CKKS-scale plaintext grows scale and requires rescale. The implementation accumulates first, sets scale manually, then rescales each result down to one modulus limb.
- Practical multiplicative depth budget: 1 for the MatMul baseline.

Sources: `FlyHE-main/FlyHE-main/RLWE/boot/src/matrix_mul.cu`, `../core/02-op-semantics.md`, `../core/04-pattern-cookbook.md`.

### 3. Packing layout and rotations

Baseline packing:

- `MM_LOG_N = 13`, `MM_N = 8192`, so CKKS has up to `4096` slots.
- Each output column ciphertext carries 4096 row results, matching the 4096 rows of `A`.
- `A` is transposed to shape `768 x 4096`, then each `A[j]` vector is encoded as one plaintext.
- `B^T` is flattened and packed into `row_pack`, one length-8192 vector per compressed ciphertext; `768 * 64 / 8192 = 6` compressed ciphertexts.

Rotation/Galois set:

- `nn.cu` builds `galois_elts` for `i = 0..MM_LOG_N-1`:

```text
galois_elt_i = (MM_N + 2^i) / 2^i
```

- `decompress_ciphertext` uses these automorphisms plus negacyclic coefficient shifts. This is not the usual slot-rotation-only diagonal method; it mixes Galois automorphism and coefficient-domain shift.
- The exact generated key set is the `galois_elts` vector passed to `create_galois_keys_from_elts`.

Sources: `FlyHE-main/FlyHE-main/RLWE/boot/examples/nn.cu`, `FlyHE-main/FlyHE-main/RLWE/boot/src/matrix_mul.cu`.

### 4. Parameters and scale schedule

Local MatMul parameters:

```text
N = 8192
slots = 4096
scale = 2^40
coeff_moduli = {60, 40, 60} for MatMul if TEST_TARGET_IDX is set to 0
```

Important local-source issue:

- `RLWE/boot/examples/nn.cu` currently sets `TEST_TARGET_IDX = 3`, which selects the LayerNorm modulus chain, not MatMul. For this MatMul experiment, change it to `0` or add a dedicated executable/config flag. This is a required source edit in the writable experiment copy, not in the HEart reference.
- `RLWE/boot/src/main.cu` sets `TEST_TARGET_IDX = 4`; its `MM_test` is similar but that file is not registered as a standalone CMake executable.

Scale/level schedule for the baseline:

```text
B compressed ct:
  level = initial chain, scale = 1e10 in enc_compress_ciphertext

B expanded ct:
  depth unchanged; scale stays from compressed ct unless custom transform changes metadata

A plaintext vector:
  encoded at scale = 2^40

multiply_plain(B_jc, A_j):
  raw scale ~= 1e10 * 2^40
  level unchanged immediately after raw multiply

add_many over j=0..767:
  all terms must have compatible level/scale
  implementation sets res_col_ct.scale() = temp_cts[0].scale()

post accumulation:
  implementation sets res_col_ct.scale() *= 4096
  then repeatedly calls rescale_to_next_inplace until one modulus limb remains
```

Risk: this path manually mutates `scale()` and relies on custom compression/decompression semantics. Before using new dimensions or publishing numbers, add assertions/logging for `chain_index`, `coeff_modulus_size`, `scale`, `is_ntt_form`, and max slotwise error after each major stage. Sources: `FlyHE-main/FlyHE-main/RLWE/boot/src/matrix_mul.cu`, `../core/03-invariants-gotchas.md`, `../libs/phantom-flyhe.md`.

Security:

- The local note does not prove 128-bit security for `N = 8192` and `{60,40,60}` plus evaluation keys. Run a lattice-estimator check for the exact secret distribution, coefficient modulus, and special/key-switching modulus before treating the parameter set as more than a benchmark [need manual confirmation]. Source: `../core/05-params-playbook.md`.

Bootstrap:

- Do not bootstrap for this baseline. The MatMul circuit is shallow, and FlyHE bootstrap examples were adjusted for A100 memory. On an 8GB RTX 3070 Laptop GPU, bootstrapping and large Galois sets are likely memory-constrained [need manual confirmation by actual run].

### 5. Required keys

Required for the current MatMul path:

- Secret key and public key.
- Relinearization key is generated by the example but not structurally required by plaintext-ciphertext MatMul.
- Galois keys for all `galois_elts` used by decompression.

Not required unless extending the pipeline:

- Bootstrap keys.
- Conversion keys.
- Conjugation key, unless a later complex/real extraction step is added.

Sources: `FlyHE-main/FlyHE-main/RLWE/boot/examples/nn.cu`, `../core/02-op-semantics.md`, `../libs/phantom-flyhe.md`.

## GPU Budget

The local session GPU check reported an NVIDIA GeForce RTX 3070 Laptop GPU with 8192 MiB total VRAM and about 4395 MiB already in use. This leaves little room for large bootstrap experiments.

Approximate ciphertext memory:

```text
bytes(ct) ~= components * active_primes * N * 8
```

For `N = 8192` and 3 active primes:

```text
2-component ct ~= 2 * 3 * 8192 * 8 = 393216 bytes ~= 0.375 MiB
3-component ct ~= 3 * 3 * 8192 * 8 = 589824 bytes ~= 0.563 MiB
```

MatMul peak objects are much larger than one ciphertext:

- 6 compressed ciphertexts for `B`.
- Up to 49152 expanded ciphertext slots in the current implementation if all decompressed ciphertexts are retained (`6 * 8192`), which is not viable at the rough ciphertext size above.
- 768 plaintext vectors for `A`.
- 64 output ciphertexts.
- Galois keys for the decompression automorphisms.
- Temporary ciphertext vector `temp_cts(768)` per output column.

Decision rule: the current implementation's all-at-once `b_expanded_cts.insert(...)` is the first memory risk. For a real large-batch run, stream decompression and multiplication by output column or by blocks of `B`, instead of materializing every expanded ciphertext before multiplication. Source: `FlyHE-main/FlyHE-main/RLWE/boot/src/matrix_mul.cu`.

## Transfer Plan

Expected host-device boundaries:

1. Read input matrices on host from text files.
2. Encode plaintext `A` vectors; encoder moves data to GPU.
3. Compress/encrypt `B`; `enc_compress_ciphertext` copies each packed host vector to GPU.
4. Evaluate MatMul on GPU.
5. Decrypt/decode final result ciphertexts for validation only.

Hot-path rule: do not call `print_decrypted_ct`, decode, or serialize inside the timed loop. Sources: `../core/gpu-considerations.md`, `../libs/phantom-flyhe.md`.

Current transfer hazard:

- `multiply_power_of_x` transforms a ciphertext out of NTT, copies ciphertext data from device to host, shifts coefficients on CPU, copies back to device, then transforms to NTT. The file contains a commented CUDA kernel version with a FIXME claiming a possible 2x speedup if correctly implemented. This host round-trip can erase GPU gains during decompression. Source: `FlyHE-main/FlyHE-main/RLWE/boot/src/matrix_mul.cu`.

Build from FlyHE root:

```bash
cd FlyHE-main/FlyHE-main
mkdir -p build
cd build
cmake ..
make -j
```

Then run from the build directory:

```bash
./bin/boot/examples/nn
```

Before running MatMul:

1. In `RLWE/boot/examples/nn.cu`, set `TEST_TARGET_IDX = 0`.
2. Confirm `../data/input/...` and `../data/calibration/...` resolve from `build/bin/boot/examples`; otherwise run from the expected directory or change paths.
3. Confirm CUDA compiler paths in top-level `CMakeLists.txt`: the local file hardcodes `/usr/bin/gcc-13`, `/usr/bin/g++-13`, and `/usr/local/cuda-12.4/bin/nvcc`.
4. On Windows/PowerShell, this tree may need WSL/Linux with CUDA toolkit because local `nvcc` was not found in PATH [need manual confirmation].

Expected output includes:

```text

Decompression took: ...

[MatMul] 4096x768 x 768x64 takes: ...
Average Error: ...
```

Use CUDA events or FlyHE's `CUDATimer` for GPU timing if you modify the benchmark. CPU wall-clock around async launches can be misleading.

## Validation Requirements

The current example reports only average absolute error for the first output column. For a real experiment, add:

- Slotwise max absolute error over all 64 output columns.
- Slotwise max relative error with an epsilon denominator.
- Max imaginary magnitude if complex decode is used.
- Per-stage logs for scale and chain index.
- A check that all generated Galois elements are actually used and all used elements are generated.

Plaintext reference:

```text
numpy.dot(A, B)
```

Acceptance should be based on project tolerance [need manual confirmation], not exact equality.

## Scaling Rules For New Matrix Sizes

Let `A` be `M x K`, `B` be `K x C`, with `S = N/2` usable CKKS slots.

- If `M <= S`, pack one output column in one ciphertext.
- If `M > S`, split rows into `ceil(M/S)` row blocks.
- Number of output ciphertexts is `C * ceil(M/S)`.
- Plaintext-vector count is `K * ceil(M/S)` if each row block needs one encoded `A[:,j]` vector.
- Rotation/Galois count depends on the compression/decompression scheme; regenerate it from the final packing layout, not from the old `MM_LOG_N = 13` constants.
- Avoid materializing `K*C` expanded ciphertexts when `K*C` is large; process in tiles.

For an encrypted-encrypted matrix product, redo the design: each dot product needs ciphertext-ciphertext multiplication, relinearization, rescale, and a much larger level/memory budget. Do not reuse this plaintext-ciphertext schedule.

## Immediate Pitfalls

- `TEST_TARGET_IDX` is not MatMul in `nn.cu` by default.
- The MatMul path is shallow but memory-heavy because it stores expanded ciphertexts.
- `multiply_power_of_x` currently performs host-device round trips.
- Scale is manually mutated; wrong scales may still run and fail numerically.
- The local build hardcodes Linux CUDA/GCC paths.
- FlyHE is research-oriented; do not treat this path as production without independent security and correctness review.
